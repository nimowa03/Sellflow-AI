# PRD 생성 규칙 (PRD Generation Rules)

## 목적
개발 시작 전에 **"무엇을 만들 것인가"**, **"무엇을 해야 하나"**, **"어떤 기준이 필요한가"**를 명확하게 규정하여 AI가 방향을 잃지 않고 정확한 제품을 만들 수 있도록 청사진을 제시합니다.

## 핵심 원칙
1. **쉽고 명확한 언어**: 주니어 개발자도 이해할 수 있는 수준으로 작성
2. **구체성**: 추상적 표현을 피하고 실행 가능한 구체적 내용 작성
3. **질문 우선**: PRD 작성 전 반드시 사용자의 요구/상황/목표를 명확히 하기 위한 질문 수행
4. **구조화**: 정해진 템플릿을 따라 일관성 유지

## PRD 작성 프로세스

### 1단계: 질문 단계 (Pre-PRD Questions)
PRD를 작성하기 전에 **반드시** 아래 질문들을 통해 정보를 수집합니다.

#### 필수 질문 항목
- **2.1 핵심 문제 정의**: 이 제품을 통해 해결하고자 하는 가장 큰 고통/문제점은 무엇인가?
- **2.2 주요 사용자 그룹**: 이 시스템의 주요 사용자는 누구인가? (기술 수준, 업무 환경 포함)
- **2.3 운영 환경**: 시스템이 실행될 환경은? (클라우드/로컬, 기존 인프라 여부)
- **2.4 필수 기능 우선순위**: MVP에 반드시 포함되어야 할 상위 3-5개 기능은?
- **2.5 연동 대상 플랫폼**: 구체적으로 어떤 플랫폼/서비스와 연동이 필요한가?
- **2.6 데이터 수집 범위**: 수집할 데이터의 양, 빈도, 법적 제약사항은?
- **2.7 AI/LLM 사용 전략**: 비용 vs 품질 우선순위, 언어 성능 요구사항은?
- **2.8 자동화 수준**: 완전 자동화 vs 반자동화? 사람의 개입 시점은?
- **2.9 성공 지표 (KPI)**: 성공을 판단할 수 있는 구체적 지표는?
- **2.10 예산 및 타임라인**: 개발/운영 예산과 완료 기한은?
- **2.11 보안 및 데이터 관리**: API 키, 데이터 저장 정책은?
- **2.12 확장성 요구사항**: 초기 규모 vs 향후 예상 규모는?

**중요**: 질문은 번호를 붙이고(2.1, 2.2...) 논리적 순서로 나열합니다.

### 2단계: 답변 수집 및 분석
사용자로부터 모든 질문에 대한 답변을 받은 후, 답변 내용을 종합 분석합니다.

### 3단계: PRD 작성
수집된 정보를 바탕으로 아래 템플릿에 따라 PRD를 작성합니다.

## PRD 템플릿 구조

### 1. 프로젝트 개요 (Project Overview)
- 프로젝트 이름
- 한 문장 요약
- 배경 및 동기

### 2. 목표 (Goals & Objectives)
- 해결하고자 하는 문제
- 기대 효과 (정량적/정성적)
- 프로젝트 성공 기준

### 3. 주요 사용자 (Target Users)
- 사용자 페르소나
- 사용자의 고충점 (Pain Points)
- 기술 수준 및 환경

### 4. 핵심 기능 (Core Features)
각 기능별로:
- **기능 설명**: 무엇을 하는가?
- **세부 내용**: 어떻게 동작하는가?
- **우선순위**: 필수(MVP) / 선택(Nice-to-have)
- **기술 요구사항**: 어떤 기술/도구가 필요한가?

### 5. 시스템 아키텍처 (System Architecture)
- 사용할 기술 스택
- 도구 및 서비스 연동 계획
- 데이터 흐름도

### 6. 비기능 요구사항 (Non-Functional Requirements)
- 성능: 응답 시간, 처리량
- 보안: 인증, 권한, 데이터 암호화
- 확장성: 향후 확장 가능성
- 유지보수성: 코드 관리, 문서화

### 7. 제약사항 (Constraints & Assumptions)
- 기술적 제약
- 비용 제약
- 시간 제약
- 법적/규제 제약

### 8. 성공 지표 (KPI & Success Metrics)
- 측정 가능한 지표 (정량적)
- 사용자 만족도 (정성적)
- 목표 수치

### 9. 프로젝트 범위 (Scope)
- 포함되는 것 (In Scope)
- 포함되지 않는 것 (Out of Scope)
- 향후 확장 계획 (Future Roadmap)

### 10. 워크플로우 흐름도 (Workflow Diagram)
- 사용자 관점 플로우
- 시스템 관점 플로우
- 주요 프로세스 단계별 설명

### 11. 릴리즈 계획 (Release Plan)
- MVP (Minimum Viable Product)
- 베타 (Beta Version)
- 정식 출시 (Production Release)

## 작성 시 주의사항

### ✅ 해야 할 것
- 쉽고 명확한 언어 사용
- 구체적인 예시 제공
- 우선순위 명확히 구분
- 측정 가능한 목표 설정
- 사용자 관점에서 작성

### ❌ 하지 말아야 할 것
- 어려운 전문 용어 남발
- 추상적이고 모호한 표현
- 우선순위 없이 나열만 하기
- 측정 불가능한 목표 설정
- 개발자 관점만 고려

## 검토 체크리스트
PRD 작성 완료 후 아래 항목을 확인합니다:

- [ ] 주니어 개발자가 읽고 이해할 수 있는가?
- [ ] 모든 필수 기능이 명확히 정의되었는가?
- [ ] 성공 기준이 측정 가능한가?
- [ ] 기술 스택과 도구가 구체적으로 명시되었는가?
- [ ] 우선순위가 명확히 구분되었는가?
- [ ] 제약사항이 명확히 기술되었는가?
- [ ] 워크플로우가 이해하기 쉽게 설명되었는가?

## 예시
```markdown
## 4. 핵심 기능

### 4.1 상품 소싱 자동화
- **기능 설명**: 쿠팡, 네이버 등 오픈마켓의 인기 상품 데이터를 자동으로 수집합니다.
- **세부 내용**:
  - 카테고리별로 상품명, 가격, 리뷰 수, 평점을 수집
  - 일 1회 자동 실행 (스케줄러 설정)
  - 수집된 데이터는 Google Sheets에 자동 저장
- **우선순위**: 필수 (MVP)
- **기술 요구사항**:
  - Apify (웹 크롤링)
  - n8n (워크플로우 자동화)
  - Google Sheets API (데이터 저장)
```

## 참고사항
- 이 규칙 파일은 모든 PRD 작성 시 참조해야 합니다
- AI 에이전트는 이 규칙을 컨텍스트로 유지하며 작업합니다
- 규칙 위반 시 즉시 수정하고 재작성합니다
